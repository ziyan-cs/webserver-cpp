# v0.2-Introduce epoll

> 完成时间：2026-05-16　对应 tag：v0.2

> 版本快照：[webserver-cpp-v0.2]()

## 阶段目标
引入 `epoll`，将服务器从单线程阻塞模型改造为事件驱动非阻塞模型，支持多客户端并发连接。

---

## 2026-05-16 | 耗时：12h（全天）

该阶段的核心任务搭建 Reactor 模型骨架，完成 `Channel`、`Epoll`、`EventLoop` 三个模块。

-  **Channel —— 单个 fd 的事件封装**
    - 定义读/写/关闭/错误 4 种回调，支持上层通过 `set*Callback()` 注入
    - `handleEvent()` 根据 `revents_` 分发对应回调
    - `events_` 只由 `enable/disableReading/Writing()` 修改，表达"想让 epoll 监听什么"；
    - `revents_` 只由 `Epoll::wait()` 写入，记录"内核实际触发了什么"
两者混用会导致 `handleEvent()` 分发逻辑出错
    - `enable/disable` 系列方法只修改内存中的 `events_`，
由上层统一调用 `update()` 同步到内核，避免每次修改都触发一次 `epoll_ctl`

- **Epoll —— epoll 系统调用封装**
    - 封装 `epoll_create / epoll_ctl / epoll_wait` 三个系统调用
    - 对外暴露 `updateEvent(Channel*)` / `delEvent(Channel*)` / `wait()` 四个接口
    - `wait()` 将内核返回的就绪事件写入对应 `Channel::revents_`，返回活跃 Channel 列表

- **EventLoop —— 主循环调度（部分实现）**
    - `loop()` 驱动主循环：调用 `epoll_.wait()` 阻塞等待，返回活跃事件列表并分发处理
    - 抽离出 `handleActiveChannels()`，逐个触发 `channel->handleEvent()`，保证 `loop()` 职责单一

**架构小结**：
1. 三个模块分工明确：`Epoll` 感知事件，`Channel` 分发回调，`EventLoop` 统一调度。
2. 上层 `Acceptor` / `TcpServer` 只需要和 `EventLoop` 交互，不接触底层实现。

## 2026-05-17 | 耗时：8h（下午）

### 实现进度

骨架跑通后暴露了两个设计问题，趁模块还少先重构，比后面依赖复杂了再动省力得多。

**问题一：`Config` 类实例传递依赖过重**

1. `Config` 原本是个类，多模块访问时需要频繁传参，每个模块都要持有一个 `Config*`，
随着模块增加耦合度越来越高。

2. 重构方案：移除 `Config` 类，将 `BUFFER_SIZE`、`MAX_EVENTS` 等核心配置改为命名空间全局变量，
`Config.h` 声明、`Config.cpp` 定义，所有模块直接通过命名空间访问，无需依赖注入。
同时将 IP / 端口信息从 `Config` 中抽离，交由新增的 `InetAddress` 类独立管理，
3. `Config` 只负责全局参数，`InetAddress` 只负责网络地址，职责边界更清晰。

**问题二：`Epoll` 私有成员导致外部类无法注册事件**

1. `Epoll` 作为 `EventLoop` 的私有成员，外部类（如 `TcpConnection`）无法直接调用
其 `add/mod/del` 接口，事件注册流程被卡住。

2. 重构方案：在 `EventLoop` 上新增 `addChannel / modChannel / delChannel` 三个公共接口，
内部转发给 `Epoll` 的对应方法。外部类只通过 `EventLoop` 操作事件，完全不接触 `Epoll` 实例。
3. Reactor 模型中 `EventLoop` 本就应该是唯一的事件入口，这次重构把这个边界显式确立下来。

**本日完成模块**

- `Config` 重构，跨模块配置共享问题解决
- `InetAddress` 类，独立管理 IP / 端口信息
- `EventLoop` 接口封装，`addChannel / modChannel / delChannel` 对外可用
- `Acceptor` 核心逻辑实现，服务端监听 socket 创建与连接接收基础流程跑通，新连接事件可正常触发

下一步目标：`TcpConnection` 接管每条连接的生命周期，接入 `ConnectionCallback` 和 `MessageCallback`。
