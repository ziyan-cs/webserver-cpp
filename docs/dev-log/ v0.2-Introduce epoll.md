# v0.2-Introduce epoll

> 完成时间：2026-05-16　对应 tag：v0.2

> 版本快照：[webserver-cpp-v0.2]()

## 阶段目标
引入 `epoll`，转变为单线程、非阻塞 IO 模型，支持多客户端并发

---

## 2026-05-16 | 耗时：10h（全天）

### 实现进度
1. **Channel 事件分发**
    - 定义读/写/关闭/错误 4 种事件回调，支持上层通过 `set*Callback()` 注入
    - `handleEvent()` 根据 `revents_` 分发对应回调
    - `enable/disableReading/Writing()` 修改内存中的 `events_`，`update()` 统一同步到内核

2. **Epoll 事件循环**
    - 封装 `epoll_create / epoll_ctl / epoll_wait` 三个系统调用
    - `updateEvent(Channel*)` / `delEvent(Channel*)` 管理 fd 的注册与移除
    - `wait()` 将内核返回的就绪事件写入对应 `Channel::revents_`，返回活跃 Channel 列表

3. **EventLoop 主循环调度**
    - `loop()` 驱动主循环：调用 `epoll_.wait()` 阻塞等待，返回活跃事件列表并分发处理
    - 抽离出 `handleActiveChannels()`，逐个触发 `channel->handleEvent()`，保证 `loop()` 职责单一
    - 对上层 Acceptor / TcpServer 暴露统一调度接口，不感知底层 epoll 细节

### 遇到问题
1. **Channel 中 `events_` 与 `revents_` 职责混淆**
    - 现象：没有分清“要监听的事件”和“内核触发的事件”
    - 解决：`events_` 只由 `enable/disable` 修改，`revents_` 只由 `Epoll::wait()` 写入

2. **纠结 `update()` 调用时机**
    - 现象：在 `enableReading()` 里 `update()` ，每次都触发 `epoll_ctl`，调用次数多
    - 解决：`enable/disable` 仅修改内存，由上层统一 `update()`，以减少系统开支

## 开发心得

本阶段完成了 Reactor 模型的核心骨架，整体分为三层。

- **底层：I/O 事件感知**
    - `Epoll` 封装系统调用，通过 `updateEvent / delEvent` 管理 fd 注册，
    - `wait()` 返回就绪的 `Channel*` 列表。
    - `Channel` 封装单个 fd 的事件语义：`events_` 表达"想监听什么"，
    - `revents_` 记录"内核触发了什么"，`handleEvent()` 据此分发四种回调，自身不感知任何业务。

- **中层：事件调度**
    - `EventLoop::loop()` 驱动主循环，调用 `epoll_.wait()` 阻塞等待，
    - 返回活跃 Channel 列表后交给 `handleActiveChannels()` 逐个触发 `handleEvent()`。
    - EventLoop 只做调度，不碰任何业务逻辑。

- **上层：连接管理（开发中）**
    - `Acceptor` 将监听 fd 包装为 Channel，`readCallback` 绑定 `handleAccept()`，
    - 新连接到来时 `accept()` 得到 `client_fd`，通过 `AcceptCallback` 向上抛给 TcpServer。
    - 后续 `TcpConnection` 将接管每条连接的生命周期，
    - 对外暴露 `ConnectionCallback` 和 `MessageCallback` 两个业务接口。

**整体做到：下层感知事件、中层调度分发、上层处理业务**
